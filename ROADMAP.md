# Roadmap

Запланированные, но ещё не реализованные части приложения `auth`.
Подробности — в `auth-app-plan.md` / `auth-app-impl.md` в корне репозитория.

## Плагины

Сами плагины (auth-провайдеры, challenge, guard, captcha) — отдельные репозитории,
заводятся и трекаются отдельно. Здесь — только релизный гейт для этого репозитория:

- [ ] Перед релизом должен быть готов и подключён хотя бы один реальный плагин
      каждого типа (`is_auth`/oauth, `is_challenge`, `is_guard`, `is_captcha`) —
      чтобы убедиться, что вся плагинная система (`authPluginManager`, конфиг
      `login_methods`/`challenge_methods`/`guard_plugins`/`captcha_plugin`)
      работает целиком, а не только с `testguard`
- [ ] Страницы настроек плагинов в backend `auth` ещё не продуманы. Сейчас
      `authBackendPlugins`/`authBackendSettings` покрывают только выбор
      auth-метода/challenge/guard/captcha (`login_methods` и т.п.), но у плагина
      могут быть свои настройки, не связанные напрямую с авторизацией (например,
      параметры подключения LDAP-сервера, ключи reCAPTCHA). Нужно посмотреть, как
      это устроено в других Webasyst-приложениях (стандартный механизм настроек
      плагина через `waPlugin::getSettings()`/`settings.html`), и сделать аналогично

## Backend

- [ ] Добавить нормальный роутинг для backend-модулей (`authBackendSettings`,
      `authBackendPlugins`, `authBackendSidebar` и т.п.) вместо вызова через
      GET-параметры (`?module=settings&action=...`)

## Уточнить

- [ ] Нужен ли отдельный `authFrontendLayout` (шаг 5 плана), или эту роль уже закрывает тема `themes/default` (head/header/footer/main)
- [ ] Восстановление пароля/доступа требует отдельной проработки сценариев, а не одного
      email-based `waAuth::sendRecoveryPassword()`. У контакта может не быть email
      (только телефон — код на SMS), не быть ни email, ни телефона (recovery вообще
      недоступен — чем заменить, ручная поддержка?), может быть несколько методов входа
      одновременно (какой использовать для recovery). Нужно обсудить и заложить в
      `authFrontendRecoveryAction`/`has_recovery` до реализации, а не по факту
- [ ] Управление привязками аккаунтов требует отдельной проработки: привязка OAuth-метода
      к уже существующему (залогиненному) контакту, отвязка метода (в т.ч. защита от
      отвязки последнего способа входа), поведение при конфликте (OAuth-аккаунт уже
      привязан к другому контакту), где всё это показывать пользователю (личный кабинет?)
- [ ] Подтверждение телефона и email (если email не в статусе `confirmed`) требует
      отдельной проработки: сейчас в приложении есть только `auth_signup_confirm` для
      подтверждения email при регистрации, но нет штатного места во фреймворке, где
      хранится факт «телефон подтверждён» — похоже, придётся вводить свою модель/поле
      под это. Нужно решить, где и как это хранить, и требовать ли подтверждения при
      смене email/телефона в личном кабинете, а не только при регистрации
- [ ] Удаление аккаунта по инициативе посетителя, скорее всего, не должно быть частью
      ядра приложения — сценариев слишком много (soft/hard delete, отвязка заказов,
      грейс-период, разные требования по разным доменам/бизнесам), логичнее отдать
      плагинам. Из этого следует, что шаблонам личного кабинета (`my.profile.html`
      и т.п.) может понадобиться механизм «хуков» — точки, где плагин может вывести
      свою кнопку/блок контента (не только для удаления аккаунта, но и вообще для
      произвольных плагинских элементов на странице профиля). Нужно продумать API
      для этого до того, как появится первый такой плагин
- [ ] Нужна система логирования событий: регистрация, вход/попытки входа, сообщения
      от guard-плагинов (кого и почему заблокировали), изменение профиля, события
      от произвольных плагинов. Администратору нужно место в backend, где это видно.
      Уточнить, использовать ли штатный `waLogModel`/`logAction()` фреймворка (как
      делает core для `signup`/`waid_auth`, см. `waOAuthController::createContact()`,
      `waWebasystIDWAAuthController::authBackendUser()`) или заводить свою таблицу —
      и как плагины смогут писать в этот же лог единообразно

## Итоговая проверка (сквозной прогон)

- [ ] Регистрация email + подтверждение письмом
- [ ] Вход email + пароль (обычный POST)
- [ ] Вход email + пароль (AJAX, модал)
- [ ] Восстановление пароля
- [ ] Вход через WAID (OAuth)
- [ ] Guard блокирует регистрацию
- [ ] Выход
- [ ] Личный кабинет — смена имени и фото
- [ ] Per-domain конфиг — два домена с разными `login_methods`
